# Commit Message Assistant

このリポジトリのコミットメッセージは、**タグ + 日本語の説明** という形式で統一する。
このスキルは、変更内容を観測したうえで適切なタグを選び、日本語のコミットメッセージを作成して `git commit` を実行する。

## トリガー

ユーザーが `commit` と打ったとき、またはコミット作成を依頼したとき。

---

## タグ一覧

| タグ | 用途 |
| --- | --- |
| `feat` | 新しい機能の追加 |
| `fix` | バグの修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | 空白、フォーマット、セミコロン追加など（コードの挙動に影響しないもの） |
| `refactor` | 仕様に影響がないコード改善（リファクタリング） |
| `perf` | パフォーマンス向上関連 |
| `test` | テスト関連 |
| `chore` | ビルド、補助ツール、ライブラリ関連 |

---

## 形式

```
<tag>: <日本語の説明>
```

- タグは小文字、半角コロン + 半角スペースで区切る
- 説明は日本語で簡潔に「何をしたか／なぜしたか」を書く
- **コミットメッセージは必ず 1 行のみ**にする。本文（2 行目以降）は書かない
  - 背景や経緯を残したい場合は、コミット本文ではなく PR 説明や issue に書く
- **タグ含めて全角換算で 20〜30 文字以内に収める**（厳守）
  - 30 文字を超える場合は、より短い表現に書き換えるかコミットを分割する
  - 20 文字未満の場合は、情報が不足していないか見直す

### 例

```
feat: 計測機能をコメントアウトで追加
fix: VOICEVOX 接続失敗時のリトライ処理を修正
docs: README にセットアップ手順を追記
refactor: sockets モジュールの import 順序を整理
chore: ruff lint 対応で sockets import に noqa: E402 を追加
```

> 直近のコミットログ（`git log --oneline -n 10`）を参照すると、このリポジトリのトーン感が掴める。

## 実行手順

### 1. 現状を観測する（並列実行）

以下のコマンドを並列で実行する：

- `git status` — 変更ファイル一覧（`-uall` フラグは使わない）
- `git diff` — ステージ済み + 未ステージの差分
- `git diff --staged` — ステージ済みの差分のみ
- `git log --oneline -n 10` — 既存のコミットメッセージのトーンを確認

### 2. `/simplify` を実行する（必須）

コミット作成前に **必ず `/simplify` スキルを実行** し、変更コードの再利用性・品質・効率を点検する。

- 指摘事項が出てきた場合は、その場で修正する
- 修正後はもう一度 `git diff` を取り直し、最新の差分で以降のステップを進める
- 「指摘なし」だった場合のみ、追加修正なしで次のステップへ進む

> このステップはスキップ不可。ユーザーが急いでいてもまず `/simplify` を回す。

#### 保護対象：意図的に残してあるコメント／コメントアウト

simplify が「不要なコメント／デッドコード」として削除を提案しても、**以下に該当する箇所は絶対に消さない**（提案を無視する）。

- コメント本文に **`一時無効化`** という文字列を含むもの
  - 例: `# import time  # 時間計測機能: 一時無効化`
- そのコメントの直下／直上に紐づく **コメントアウトされたコード行**（一時無効化と一体で意味を持つため）

過去に同じコメントを誤って削除した事故があり（`07118d6 fix: 時間計測機能のコメントを誤削除から復元`）、その再発防止のためのルール。simplify の指摘が保護対象に該当する場合は、その指摘だけスキップしてユーザーに「保護対象のため除外した」と一言伝える。

### 3. タグを判定する

差分の内容から、上の表をもとに最も適切なタグを 1 つ選ぶ。判断に迷う場合：

- 機能追加とリファクタが混在 → 主たる目的が機能追加なら `feat`
- バグ修正とテスト追加が混在 → `fix`（テストはその修正の一部とみなす）
- 複数の独立した変更が混ざっている → ユーザーに分割コミットを提案する

### 4. コミットメッセージを下書きする

- 日本語で「何を／なぜ」を **1 行のみ**にまとめる（本文は付けない）
- **全角換算で 20〜30 文字以内**（タグ含む）。書き終えたら必ず文字数を数えて検算する
- 直近のコミットログのトーン・粒度に合わせる
- "What" だけでなく "Why" を意識する（例: × `chore: import 順を変更` / ○ `chore: ruff lint 対応で import 順を整理`）

### 5. ユーザーに確認する

ドラフトしたメッセージを提示し、ユーザーの承認を待つ。修正依頼があれば反映する。

### 6. 必要なファイルをステージしてコミット

- `git add` で対象ファイルを明示的に追加する（`git add -A` や `git add .` は使わない）
- `.env` や認証情報を含むファイルは絶対にコミットしない
- メッセージは 1 行なので `-m` に直接渡す：

```bash
git commit -m "<tag>: <日本語の説明>"
```

### 7. 結果を確認

`git status` でコミット成功を確認し、ユーザーに簡潔に報告する。

---

## 注意事項

- **Co-Authored-By などの署名は付けない。** このリポジトリのコミットログには付いていないため、トーンを揃える。
- **`--amend` は使わない。** 必ず新しいコミットを作成する（ユーザーが明示的に依頼した場合を除く）。
- **`--no-verify` は使わない。** pre-commit hook が失敗したら、原因を調べて修正する。
- **push は自動で行わない。** ユーザーが明示的に依頼した場合のみ実行する。
- ユーザーから依頼がない限り、コミットを勝手に作成しない。
